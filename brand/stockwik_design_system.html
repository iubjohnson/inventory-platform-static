0 · Typography

Two typefaces, each with a clear job. Rajdhani Bold is the display face — condensed, squared, and technical, it gives the brand its fast, precise feel and is especially strong on numbers. Inter is the workhorse for everything people actually read closely. The rule that keeps it coherent: Rajdhani for large or short-form display (headings, the wordmark, big metric numbers, buttons, badges); Inter for everything small or running (body copy, labels, table data, captions, code). Condensed faces cramp at small sizes, so don't reach for Rajdhani on small paragraph or label text.

Type scale

A simple scale keeps sizes consistent. Display 40 / H1 30 / H2 21 / H3 16 (Rajdhani); body 15 / small 13 / label 12 (Inter). Line-height ~1.0–1.1 for Rajdhani display (tight, since it's condensed), ~1.55–1.65 for Inter body (open, for readability).

1 · Buttons

The button hierarchy follows one rule: amber means "do this," indigo means "this is structural." Reserve the amber primary for the single most important action on a view — the rest of the page uses quieter styles so that one action stands out. Labels are Rajdhani SemiBold; corners use an 8px radius echoing the loop mark's rounded ends.

Primary — one per view, the main action
Amber #EF9F27 with ink text (#1A1640) — ink not white, since amber is too light for white text to meet contrast (7.84:1 with ink vs. 2.17:1 with white).
Secondary — important, not the hero
Indigo #26215C with white text (14.4:1).
Outline & ghost — secondary / low-stakes
Destructive — deletes and removals
Deliberately outside the brand palette so destructive actions never blend in. Soft by default; solid red for final confirmation.
Pairing & disabled state
On a dark / indigo background

Button rules

  • Use exactly one amber primary per screen or section — it marks the main action.
  • Always pair a filled primary with a quieter outline/ghost for the secondary choice.
  • Keep a visible focus ring for keyboard users (the purple outline on focus).
  • Don't put two amber primaries side by side — they cancel each other out.
  • Don't use white text on amber (fails contrast) — use ink instead.
  • Don't use the brand palette for destructive actions — keep red separate.

2 · Tabs

Tabs are navigation, not actions — so they stay quiet and don't compete with amber buttons. The default is an underline style: muted gray inactive labels, with the active tab getting darker indigo text plus a 2px amber underline as a "you are here" cue. Best for 2–4 short labels.

Optional count badges show volume before a click — useful for data tabs. Drop them where they help.

Alternate — soft pill (use sparingly)

Pill tabs read as a compact toggle — good for a small "switch the view of this one thing" control (table vs. chart), less good as primary page-section navigation, where they get heavy.

Tab rules

  • Keep the underline style for primary page-section navigation.
  • Implement proper ARIA roles (tablist / tab / tabpanel) and arrow-key navigation.
  • Make the whole tab a click target, not just the text.
  • Don't mute inactive labels so far they look disabled.
  • Don't mix underline and pill styles for the same purpose — reserve pills for inline toggles.
Implementation note. In the React/Vite app, don't hand-roll tab logic — drive the active tab with state, or use an accessible headless library (Radix UI Tabs, Headless UI) and apply these CSS classes. That gets keyboard/ARIA handling for free. The styling here is the deliverable; any inline demo script is throwaway scaffolding.

3 · Tables

Tables are the workhorse of the app, usually showing long SKU/inventory lists. Defaults are tuned for that: compact row spacing so more rows fit on screen, right-aligned tabular numbers so quantities and costs line up for easy comparison, and clickable rows with a clear hover state. Comfortable and relaxed density variants are available for shorter tables.

Default — compact, clickable rows

Vertical padding of 9px keeps long lists scannable. Rows show a pointer cursor and an indigo-soft hover tint; numbers are right-aligned in tabular figures.

Table rules

  • Right-align numeric columns and use tabular figures so digits line up.
  • Default to compact spacing for long lists; reserve relaxed for short tables.
  • Give clickable rows a pointer cursor, a hover tint, and a keyboard focus outline.
  • Use a sticky header for any list long enough to scroll.
  • Don't add vertical gridlines — horizontal hairlines alone are cleaner to scan.
  • Don't use zebra striping on narrow tables; reserve it for very wide ones where the eye loses its row.
  • Don't left-align numbers or mix proportional and tabular figures in a column.
Implementation note. For long lists in the React app, plain HTML tables are fine into the low hundreds of rows. Past that, reach for a virtualized table (TanStack Table + virtualization) so only visible rows render — but keep these exact classes for styling so the look stays consistent. Sorting, filtering, and column resizing also live in TanStack Table rather than hand-rolled.

4 · Form inputs

Forms appear on every create/edit screen — PO builders, supplier settings, product detail. The defaults: a Rajdhani SemiBold label, an Inter input at 14px, an 8px radius to match buttons, and a purple focus ring matching the rest of the system. Keep field widths consistent (≈420px max for single-column forms) so edges align.

Form rules

  • Always pair an input with a label; mark optional fields, not required ones (required is the default expectation).
  • Put units and currency in affixes ($, days, lb) so the value field stays clean.
  • Show errors inline below the field in red, with a specific message — not just a red border.
  • Keep one consistent max width so stacked fields align on the left and right.
  • Don't use placeholder text as a substitute for a label — it vanishes on input.
  • Don't reach for amber on inputs; the purple focus ring is the system's input accent.

5 · Status badges

Inventory states recur on nearly every screen, so their colors are locked semantically — the same state is always the same color, app-wide. These replace the ad-hoc pills used earlier in the table examples. The optional leading dot helps the state read at a glance in dense tables.
